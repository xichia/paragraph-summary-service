# Artifact Provenance

This document describes the provenance metadata written into generated JSONL
summary artifacts.

## Current implemented metadata

Every JSONL line in the generated artifact includes these fields
(`writer.py:_summary_to_line()`):

| Field | Type | Always present | Description |
|---|---|---|---|
| `document_id` | string | Yes | Identifier from the request. |
| `record_id` | string | Yes | Canonical record identifier. |
| `source_ref` | string | Yes | Human-readable source location. |
| `input_text_hash` | string | Yes | `sha256:` prefix of the normalized input text. |
| `summary` | string | Yes | Generated one-sentence summary (empty string on failure). |
| `summary_style` | string | Yes | Style identifier from request/template. |
| `template_version` | string | Yes | Prompt template version identifier. |
| `provider` | string | Yes | Provider name (`"mock"` or `"gemini"`). |
| `model` | string | Yes | Model name (e.g. `"mock-deterministic-v1"`, `"gemini-3.1-flash-lite"`). |
| `cache_hit` | boolean | Yes | Whether the result was served from cache. |
| `status` | string | Yes | Record-level status (`"completed"` or `"failed"`). |
| `error_code` | string or null | Yes | Error code if `status` is `"failed"`, else `null`. |
| `usage` | object | Yes | Token and cost breakdown (see below). |
| `metadata` | object or null | Depends | Redacted request metadata; may be `null`. |
| `provenance` | string | Depends | Source provenance tag (e.g. `"deepreader"`). |

### `usage` sub-fields

| Field | Type | Description |
|---|---|---|
| `input_tokens` | integer | Estimated or reported input token count. |
| `output_tokens` | integer | Estimated or reported output token count. |
| `total_tokens` | integer | Sum of input and output tokens. |
| `estimated_cost_usd` | float | Estimated cost in USD (currently `0.0` for Gemini; cost table is not yet populated). |

### Per-artifact response metadata

The API response (`ParagraphSummaryArtifactResponse`) returns artifact-level
metadata:

| Field | Type | Description |
|---|---|---|
| `artifact_id` | string | Unique artifact identifier. |
| `document_id` | string | Request document identifier. |
| `output_path` | string | Relative filesystem path to the artifact. |
| `records_received` | integer | Number of input records. |
| `records_completed` | integer | Records with `status="completed"`. |
| `records_failed` | integer | Records with `status="failed"`. |
| `cache_hits` | integer | Count of cache-served results. |
| `provider` | string | Provider used for the request. |
| `model` | string | Model used for the request. |
| `template_version` | string | Prompt template version. |
| `usage` | object | Aggregated usage across all records. |
| `batches_processed` | integer | Number of provider batches sent. |

## Proposed / future metadata

The following fields are **not yet implemented** and are recorded here as design
targets:

| Field | Rationale |
|---|---|
| `artifact_id` (in each JSONL line) | Currently only in the response envelope, not per-line. Would allow individual line provenance. |
| `runtime_mode` (`mock`/`offline`/`live`) | Would distinguish mock, offline (cached), and live provider runs. |
| `input_record_count` (per line) | Redundant with the envelope count, but useful for partial-artifact inspection. |
| `records_skipped` / `records_cancelled` | Would accompany lifecycle status expansion (see [SUMMARY_JOB_LIFECYCLE.md](SUMMARY_JOB_LIFECYCLE.md)). |
| `redaction_status` | Whether redaction was applied to the input text for this record. |
| `source_payload_fingerprint` | Hash of the full request payload for cross-reference. |
| `validation_workflow_ref` | Reference to the validation workflow or CI run that produced the artifact. |
| `model_version` | Specific model version or checkpoint, not just the model name. |

## Artifact identification

Artifact IDs are generated by `make_artifact_id()` in `artifacts/ids.py`:

```
artifact_{slugified_document_id}_{sha256_prefix_of_record_ids}_{timestamp_ms}
```

The output filename is:

```
{artifact_dir}/{sluggified_document_id}.paragraph_summaries.jsonl
```

## Schema stability

The current JSONL schema should be treated as stable within a `v0.1.x` release.
Field additions are preferred over field renames or removals. This document
labels proposed fields clearly so reviewers can distinguish current metadata
from future additions.

## Sidecar contract

Each artifact is a standalone JSONL file. DeepReader or any consumer can
iterate the file line by line. The artifact does not include an external
manifest or checksum file (not yet implemented).
